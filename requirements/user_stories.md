# User Stories – Delivery Orchestration Service

Формат: **Connextra** (`As a … I want … so that …`) + **Acceptance Criteria** в нотации
**Gherkin** (Given–When–Then). Каждая история снабжена приоритетом (MoSCoW) и оценкой
в story points.

Акторы: **Покупатель**, **Продавец (Merchant)**, **`order-service`** (системный),
**Оператор поддержки**, **Перевозчик** (внешняя система через вебхуки).

---

## US-01 – Расчёт вариантов доставки в корзине

**Приоритет:** Must · **SP:** 8

> **Как** покупатель,
> **я хочу** видеть стоимость и срок доставки по доступным способам ещё до оформления,
> **чтобы** выбрать подходящий вариант и не бросить корзину из-за неожиданной цены.

### Acceptance Criteria

```gherkin
Feature: Расчёт вариантов доставки

  Background:
    Given в корзине есть товары от продавца "ACME" со склада "MSK-01"
    And покупатель указал город доставки "Казань" и индекс "420012"

  Scenario: Успешный расчёт по нескольким перевозчикам
    When система запрашивает расчёт доставки
    Then в ответе присутствует минимум один вариант от перевозчика "СДЭК"
    And минимум один вариант от перевозчика "Boxberry"
    And для каждого варианта указаны: стоимость в рублях, срок в днях, способ (ПВЗ/курьер/постамат)
    And ответ возвращается не дольше чем за 700 мс по p95

  Scenario: Один из перевозчиков недоступен – graceful degradation
    Given API перевозчика "Boxberry" отвечает с ошибкой или таймаутом
    When система запрашивает расчёт доставки
    Then в ответе всё равно присутствуют варианты от доступных перевозчиков
    And для недоступного перевозчика возвращается пометка "temporarily_unavailable"
    And HTTP-статус ответа равен 200

  Scenario: Доставка в город невозможна ни одним перевозчиком
    Given ни один перевозчик не обслуживает индекс "999999"
    When система запрашивает расчёт доставки
    Then возвращается пустой список вариантов
    And код причины равен "NO_DELIVERY_OPTIONS"
    And HTTP-статус ответа равен 200
```

## US-02 – Сплит заказа на отгрузки

**Приоритет:** Must · **SP:** 13

> **Как** покупатель,
> **я хочу** оформить единый заказ с товарами от разных продавцов,
> **чтобы** не создавать отдельные заказы вручную, при этом видеть доставку по частям.

### Acceptance Criteria

```gherkin
Feature: Сплит заказа на отгрузки

  Scenario: Заказ с товарами двух продавцов разбивается на две отгрузки
    Given заказ содержит товар A продавца "ACME" со склада "MSK-01"
    And товар B продавца "Globex" со склада "SPB-02"
    When система выполняет сплит заказа
    Then создаётся ровно 2 отгрузки (shipment)
    And отгрузка 1 содержит только товар A и привязана к складу "MSK-01"
    And отгрузка 2 содержит только товар B и привязана к складу "SPB-02"
    And сумма стоимостей отгрузок равна итоговой стоимости доставки заказа

  Scenario: Превышение габаритов одной посылки внутри одного продавца
    Given заказ содержит 10 единиц крупногабаритного товара продавца "ACME"
    And максимальный вес одной посылки перевозчика равен 30 кг
    And суммарный вес товаров равен 80 кг
    When система выполняет сплит заказа
    Then товары распределяются минимум на 3 отгрузки одного продавца
    And ни одна отгрузка не превышает 30 кг

  Scenario: Идемпотентность сплита
    Given заказ "ORD-100" уже был успешно разбит на отгрузки
    When система повторно получает команду сплита для "ORD-100" с тем же ключом идемпотентности
    Then новые отгрузки не создаются
    And возвращается ранее созданный набор отгрузок
```

## US-03 – Оформление доставки и получение трек-номеров

**Приоритет:** Must · **SP:** 8

> **Как** `order-service`,
> **я хочу** после оплаты заказа закрепить выбранные способы доставки и получить
> трек-номера,
> **чтобы** передать заказ продавцам на сборку и показать покупателю отслеживание.

### Acceptance Criteria

```gherkin
Feature: Оформление доставки

  Scenario: Успешное создание заказов у перевозчиков
    Given заказ "ORD-100" оплачен и разбит на 2 отгрузки
    When order-service подтверждает доставку
    Then для каждой отгрузки создаётся заказ в системе соответствующего перевозчика
    And каждая отгрузка получает трек-номер (tracking_number)
    And статус каждой отгрузки переходит в "CREATED"
    And публикуется событие "shipment.created" для каждой отгрузки

  Scenario: Сбой создания у перевозчика по одной из отгрузок
    Given заказ "ORD-100" разбит на 2 отгрузки
    And перевозчик отгрузки 2 возвращает ошибку при создании заказа
    When order-service подтверждает доставку
    Then отгрузка 1 переходит в статус "CREATED"
    And отгрузка 2 переходит в статус "CREATION_FAILED"
    And создаётся задача на повторную попытку с экспоненциальным backoff
    And событие "shipment.creation_failed" публикуется для отгрузки 2
```

## US-04 – Отслеживание статуса доставки в реальном времени

**Приоритет:** Must · **SP:** 8

> **Как** покупатель,
> **я хочу** видеть актуальный статус каждой посылки моего заказа,
> **чтобы** понимать, когда забрать товар, и не обращаться лишний раз в поддержку.

### Acceptance Criteria

```gherkin
Feature: Синхронизация статусов по вебхукам

  Scenario: Обновление статуса по вебхуку перевозчика
    Given отгрузка "SHP-1" имеет статус "IN_TRANSIT"
    When DOS получает вебхук от перевозчика со статусом "DELIVERED_TO_PVZ" для "SHP-1"
    Then внутренний статус "SHP-1" переходит в "READY_FOR_PICKUP"
    And покупателю доступно событие обновления статуса не позднее 90 секунд (p95)
    And публикуется событие "shipment.status_changed"

  Scenario: Дедупликация повторного вебхука
    Given вебхук с external_event_id "EVT-555" уже был обработан
    When DOS повторно получает вебхук с тем же external_event_id "EVT-555"
    Then статус отгрузки не меняется повторно
    And повторное доменное событие не публикуется
    And DOS отвечает перевозчику HTTP 200

  Scenario: Недопустимый переход статуса
    Given отгрузка "SHP-1" уже в статусе "DELIVERED"
    When DOS получает вебхук со статусом "IN_TRANSIT"
    Then переход отклоняется как недопустимый
    And инцидент логируется для разбора
    And текущий статус "DELIVERED" сохраняется
```

## US-05 – Отмена доставки покупателем

**Приоритет:** Should · **SP:** 13

> **Как** покупатель,
> **я хочу** отменить доставку отдельной посылки до её получения,
> **чтобы** не ехать за ненужным товаром и инициировать возврат средств.

### Acceptance Criteria

```gherkin
Feature: Отмена отгрузки покупателем

  Scenario: Отмена отгрузки, ещё не переданной перевозчику
    Given отгрузка "SHP-1" в статусе "CREATED"
    When покупатель запрашивает отмену "SHP-1"
    Then заказ у перевозчика аннулируется
    And статус "SHP-1" переходит в "CANCELLED"
    And публикуется событие "shipment.cancelled" с причиной "customer_request"

  Scenario: Отмена при недоступности API перевозчика (Circuit Breaker открыт)
    Given отгрузка "SHP-1" в статусе "IN_TRANSIT"
    And предохранитель (circuit breaker) на API перевозчика находится в состоянии OPEN
    When покупатель запрашивает отмену "SHP-1"
    Then статус "SHP-1" переходит в "CANCELLATION_PENDING"
    And команда отмены ставится в очередь на повтор
    And покупателю сообщается, что отмена обрабатывается
    And после восстановления связи отмена повторяется автоматически

  Scenario: Отмену запрещено выполнять после получения
    Given отгрузка "SHP-1" в статусе "DELIVERED"
    When покупатель запрашивает отмену "SHP-1"
    Then запрос отклоняется с кодом "CANCELLATION_NOT_ALLOWED"
    And статус "SHP-1" не меняется
```

## US-06 – Управление складами и габаритами продавцом

**Приоритет:** Could · **SP:** 5

> **Как** продавец,
> **я хочу** задавать склады отгрузки и весогабаритные характеристики товаров,
> **чтобы** система корректно считала доставку и сплитила заказы.

### Acceptance Criteria

```gherkin
Feature: Управление параметрами отгрузки продавцом

  Scenario: Указание склада отгрузки для товара
    Given продавец "ACME" имеет склады "MSK-01" и "EKB-03"
    When продавец назначает товару "SKU-777" склад отгрузки "EKB-03"
    Then последующие расчёты доставки этого товара используют склад "EKB-03" как точку отправления

  Scenario: Валидация весогабаритных параметров
    Given продавец задаёт товару вес 0 кг или отрицательные габариты
    When продавец сохраняет параметры товара
    Then сохранение отклоняется с ошибкой валидации "INVALID_DIMENSIONS"
    And товар не участвует в расчёте доставки до исправления данных
```

## Нефункциональные требования (NFR)

| ID | Категория | Требование |
|------|-----------|------------|
| NFR-01 | Производительность | p95 `POST /delivery/calculate` ≤ 700 мс при 500 RPS |
| NFR-02 | Доступность | SLA сервиса 99,9 %; недоступность одного перевозчика не роняет расчёт |
| NFR-03 | Отказоустойчивость | Circuit Breaker + Retry с backoff на всех внешних вызовах |
| NFR-04 | Идемпотентность | Все мутирующие операции принимают `Idempotency-Key` |
| NFR-05 | Согласованность | Доставка событий at-least-once через Transactional Outbox |
| NFR-06 | Наблюдаемость | Сквозной trace_id во всех логах и событиях; дашборды p50/p95/p99 |
| NFR-07 | Безопасность | Подпись вебхуков перевозчиков проверяется (HMAC); mTLS внутри кластера |
| NFR-08 | Масштабируемость | Stateless-инстансы, горизонтальное масштабирование в Kubernetes |
