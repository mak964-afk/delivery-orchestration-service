# Use Case Specifications – Delivery Orchestration Service

Спецификации ключевых вариантов использования. Формат – расширенный (Cockburn):
основной (happy path), альтернативные и исключительные сценарии, пред-/постусловия.

## Акторы

| Актор | Тип | Роль |
|-------|-----|------|
| Покупатель | Первичный, человек | Выбирает доставку, отслеживает, отменяет |
| Продавец (Merchant) | Первичный, человек | Задаёт склады и габариты, собирает отгрузки |
| `order-service` | Первичный, система | Инициирует расчёт и оформление доставки |
| Оператор поддержки | Вторичный, человек | Разбирает инциденты, ручная отмена |
| СДЭК / Boxberry | Вторичный, внешняя система | Создаёт заказы, шлёт статусы по вебхукам |
| `payment-service` | Вторичный, система | Подтверждает оплату (триггер оформления) |

## UC-01 – Рассчитать стоимость доставки

| Поле | Значение |
|------|----------|
| **ID** | UC-01 |
| **Актор (первичный)** | `order-service` (по инициативе Покупателя в корзине) |
| **Цель** | Получить список доступных вариантов доставки со стоимостью и сроком |
| **Триггер** | Покупатель указал адрес/город доставки в корзине |
| **Предусловия** | Корзина непуста; у товаров заданы склад отправления и весогабариты |
| **Частота** | Высокая (десятки запросов на оформление, при каждом изменении адреса) |

### Основной сценарий (Main Success Scenario)

1. `order-service` отправляет в DOS состав корзины и пункт назначения.
2. DOS группирует позиции по `(продавец, склад)` – формирует предварительные группы отгрузок.
3. DOS для каждой группы определяет применимых перевозчиков и способы доставки.
4. DOS запрашивает тарифы у адаптеров перевозчиков (параллельно, с таймаутом 600 мс).
5. DOS агрегирует ответы, суммирует стоимость по группам, формирует варианты заказа.
6. DOS возвращает `order-service` список вариантов (стоимость, срок, способ, перевозчик).

### Альтернативные сценарии

- **3a. Тариф найден в кэше Redis:** DOS пропускает шаг 4 для этой группы и берёт
  закэшированный тариф (TTL 15 мин), снижая latency.
- **4a. Часть перевозчиков не ответила в таймаут:** DOS исключает их из выдачи,
  помечает `temporarily_unavailable`, продолжает с доступными (шаг 5).

### Исключительные сценарии

- **4b. Все перевозчики недоступны, кэш пуст:** DOS возвращает `503` с кодом
  `ALL_CARRIERS_UNAVAILABLE`; `order-service` показывает покупателю мягкую ошибку и
  предлагает повторить позже.
- **3b. Ни один перевозчик не обслуживает направление:** DOS возвращает пустой список
  и код `NO_DELIVERY_OPTIONS` (HTTP 200).

### Постусловия

- **Успех:** покупателю показаны актуальные варианты; состояние системы не изменено
  (расчёт – операция чтения, без побочных эффектов, кроме записи в кэш).
- **Неуспех:** заказ не создан; зафиксирована метрика недоступности перевозчиков.

## UC-02 – Оформить доставку заказа (сплит + создание у перевозчиков)

| Поле | Значение |
|------|----------|
| **ID** | UC-02 |
| **Актор (первичный)** | `order-service` |
| **Цель** | Разбить заказ на отгрузки и создать заказы у перевозчиков, получить треки |
| **Триггер** | Событие `order.paid` от `payment-service` |
| **Предусловия** | Заказ оплачен; выбран вариант доставки; адрес валиден |

### Основной сценарий

1. DOS получает команду оформления доставки с `Idempotency-Key`.
2. DOS проверяет, не оформлялась ли доставка ранее по этому ключу.
3. DOS выполняет сплит заказа на отгрузки по правилам (продавец, склад, габариты).
4. DOS сохраняет отгрузки в статусе `PENDING` (в одной транзакции с записью в outbox).
5. Для каждой отгрузки DOS вызывает адаптер перевозчика для создания заказа.
6. DOS сохраняет трек-номер, переводит отгрузку в `CREATED`.
7. DOS публикует событие `shipment.created` по каждой отгрузке.

### Альтернативные сценарии

- **2a. Ключ идемпотентности уже использован:** DOS возвращает ранее созданный набор
  отгрузок без побочных эффектов.
- **5a. Перевозчик вернул ошибку по одной отгрузке:** отгрузка переводится в
  `CREATION_FAILED`, ставится задача повтора (Retry с backoff); остальные отгрузки
  оформляются штатно (частичный успех).

### Исключительные сценарии

- **5b. Все попытки создания исчерпаны:** отгрузка остаётся `CREATION_FAILED`,
  публикуется `shipment.creation_failed`, эскалация Оператору поддержки.
- **3a. Сплит невозможен (нет валидного склада):** оформление прерывается, заказ
  помечается `DELIVERY_SETUP_FAILED`, событие уходит в `order-service`.

### Постусловия

- **Успех:** все отгрузки в `CREATED`, треки получены, события опубликованы.
- **Частичный успех:** ≥ 1 отгрузка `CREATED`, проблемные – в ретрае.
- **Неуспех:** заказ не передан в логистику; инцидент заведён.

## UC-03 – Отменить отгрузку (инициатор – Покупатель)

| Поле | Значение |
|------|----------|
| **ID** | UC-03 |
| **Актор (первичный)** | Покупатель |
| **Цель** | Аннулировать доставку посылки до получения |
| **Триггер** | Покупатель нажимает «Отменить» по конкретной отгрузке |
| **Предусловия** | Отгрузка существует и в статусе, допускающем отмену |

### Основной сценарий

1. Покупатель инициирует отмену отгрузки `SHP-1`.
2. DOS проверяет, что текущий статус допускает отмену (`CREATED`/`IN_TRANSIT`).
3. DOS вызывает адаптер перевозчика для аннулирования заказа.
4. Перевозчик подтверждает аннулирование.
5. DOS переводит отгрузку в `CANCELLED` (причина `customer_request`).
6. DOS публикует `shipment.cancelled` (триггер возврата средств в `payment-service`).

### Альтернативные сценарии

- **3a. API перевозчика недоступно (Circuit Breaker = OPEN):** DOS переводит отгрузку в
  `CANCELLATION_PENDING`, ставит команду в очередь повтора, сообщает покупателю «отмена
  обрабатывается»; повтор выполняется по восстановлении связи.
- **4a. Перевозчик ответил «уже передано курьеру»:** DOS уведомляет покупателя, что
  отмена невозможна на текущем этапе, предлагает оформить возврат после получения.

### Исключительные сценарии

- **2a. Статус `DELIVERED`/`CANCELLED`:** отмена отклоняется с
  `CANCELLATION_NOT_ALLOWED`; состояние не меняется.

### Постусловия

- **Успех:** отгрузка `CANCELLED`, инициирован возврат, событие опубликовано.
- **Отложено:** отгрузка `CANCELLATION_PENDING`, отмена в очереди.

## UC-04 – Обработать вебхук статуса от перевозчика

| Поле | Значение |
|------|----------|
| **ID** | UC-04 |
| **Актор (первичный)** | СДЭК / Boxberry (внешняя система) |
| **Цель** | Синхронизировать внутренний статус отгрузки с фактическим |
| **Триггер** | HTTP-вебхук от перевозчика об изменении статуса |
| **Предусловия** | Отгрузка существует; подпись вебхука валидна |

### Основной сценарий

1. Перевозчик отправляет вебхук с `external_event_id` и новым статусом.
2. DOS проверяет HMAC-подпись запроса.
3. DOS проверяет, не обрабатывался ли `external_event_id` ранее (дедупликация).
4. DOS мапит внешний статус перевозчика во внутренний (через ACL).
5. DOS проверяет допустимость перехода по конечному автомату статусов.
6. DOS обновляет статус, пишет событие в outbox, отвечает перевозчику `200`.
7. Publisher публикует `shipment.status_changed` в Kafka.

### Альтернативные / исключительные сценарии

- **2a. Подпись невалидна:** запрос отклоняется `401`, инцидент безопасности логируется.
- **3a. Дубль события:** DOS отвечает `200` без изменения состояния (идемпотентность).
- **5a. Недопустимый переход:** статус не меняется, событие логируется для разбора,
  ответ `200` (чтобы перевозчик не ретраил бесконечно).

### Постусловия

- **Успех:** статус актуализирован, событие опубликовано, лаг ≤ 90 c (p95).
- **Идемпотентный дубль:** состояние неизменно.
